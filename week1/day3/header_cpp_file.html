üîπ What is a Header File?

Simple Definition:

A header file is a file that contains declarations and ready-made code that we can reuse in our programs. Think of it as a library of pre-written functionality.

File Extensions

.h    // C-style header
.hpp  // C++ style header

Common Examples

#include <iostream>  // For input/output operations
#include <cmath>      // For mathematical functions
#include <vector>     // For dynamic arrays
#include <string>     // For string manipulation

What Header Files Contain

  • Function Declarations: Prototypes that tell the compiler about functions
  • Function Definitions: Actual implementation of functions
  • Class Declarations: Blueprints for objects
  • Data Type Definitions: Custom types and structures
  • Macros: Text replacements for code reusability
  • Constants: Global constant values

üîπ Standard Header Files

These are built-in header files that come with the C++ standard library. They provide commonly used functions and classes so you don't have to write everything from scratch.

<iostream> - Input/Output Streams

Use Case:

Console input/output operations

Provides:

  • std::cout - Output to console
  • std::cin - Input from console
  • std::endl - End line and flush buffer
  • std::cerr - Error output stream
#include <iostream>
using namespace std;

int main() {
    cout << "Hello, Deepak!" << endl;
    
    string name;
    cout << "Enter your name: ";
    cin >> name;
    cout << "Welcome, " << name << endl;
    
    return 0;
}

<cmath> - Mathematical Functions

Common Functions:

  • sqrt(x) - Square root
  • pow(base, exponent) - Power calculation
  • abs(x) - Absolute value
  • ceil(x) - Round up
  • floor(x) - Round down
  • sin(x), cos(x), tan(x) - Trigonometric functions
  • log(x), log10(x) - Logarithmic functions

}
Square root of 25: 5 2^3: 8 Absolute of -42: 42

üîπ Understanding #include Directive

What is #include?

#include is a preprocessor directive. It works BEFORE compilation begins.

How It Works

1

Preprocessing Stage

When you write #include <iostream>, the preprocessor literally copies the entire contents of the iostream file and pastes it into your source file.

2

File Replacement

// Your code:
#include <iostream>

int main() {
    std::cout << "Hello";
}

// After preprocessing (simplified):
// [Entire iostream header content pasted here - thousands of lines]

int main() {
    std::cout << "Hello";
}
3

Then Compilation Begins

Only after all #include directives are processed does the actual compilation start.

Important:

#include is NOT a C++ statement - it's a preprocessor command. It doesn't end with a semicolon!

üîπ Types of Include

Syntax Type Search Location
#include <file> Standard Library System/compiler directories
#include "file" User-Defined Current directory first, then system

Angle Brackets < > - Standard Headers

#include <iostream>
#include <vector>
#include <cmath>

The compiler searches in standard system directories (like /usr/include on Linux or compiler installation folders).

Double Quotes " " - User-Defined Headers

#include "mathutils.h"
#include "myclass.h"
#include "config.h"

The compiler first searches in the current directory. If not found, it searches in system directories.

Best Practice:

Use angle brackets for standard library headers and double quotes for your own headers.

üîπ User-Defined Header Files

Creating your own header files helps organize code, improves reusability, and makes large projects manageable.

Complete Example: Creating mathutils Header

1

Create mathutils.h (Header File)

This file contains function declarations (prototypes).

// mathutils.h
// If MATHUTILS_H is not defined, define it (start of header guard)
#ifndef MATHUTILS_H
#define MATHUTILS_H

// Function declarations (only the function names and parameters)
int add(int a, int b);
int subtract(int a, int b);
int multiply(int a, int b);
double divide(double a, double b);
int power(int base, int exponent);

#endif // MATHUTILS_H
Why only declarations?

The header file acts as an interface - it tells other files what functions are available, but not how they work.

2

Create mathutils.cpp (Implementation File)

This file contains function definitions (actual logic).

// mathutils.cpp
#include "mathutils.h"

// Function definitions
int add(int a, int b) {
    return a + b;
}

int subtract(int a, int b) {
    return a - b;
}

int multiply(int a, int b) {
    return a * b;
}

double divide(double a, double b) {
    if (b != 0) {
        return a / b;
    }
    return 0; // Handle division by zero
}

int power(int base, int exponent) {
    int result = 1;
    for (int i = 0; i < exponent; i++) {
        result *= base;
    }
    return result;
}
3

Create main.cpp (Main Program)

This file uses the functions from your header.

// main.cpp
#include <iostream>
#include "mathutils.h"  // Include our header

using namespace std;

int main() {
    cout << "Add: " << add(5, 3) << endl;
    cout << "Subtract: " << subtract(10, 4) << endl;
    cout << "Multiply: " << multiply(4, 6) << endl;
    cout << "Divide: " << divide(20.0, 4.0) << endl;
    cout << "Power: " << power(2, 5) << endl;
    
    return 0;
}
File Structure: 
project/
‚îú‚îÄ‚îÄ mathutils.h      (Declarations)
‚îú‚îÄ‚îÄ mathutils.cpp    (Definitions)
‚îî‚îÄ‚îÄ main.cpp         (Main program)

üîπ Include Guards - Critical Concept

The Problem: Multiple Inclusion

What happens if a header is included multiple times?

Consider this scenario:

// utils.h
struct Point {
    int x, y;
};

// geometry.h
#include "utils.h"

// main.cpp
#include "utils.h"
#include "geometry.h"

After preprocessing, main.cpp becomes:

// First include of utils.h
struct Point {
    int x, y;
};

// Second include via geometry.h
struct Point {  // ERROR! Redefinition!
    int x, y;
};
Error: redefinition of 'struct Point'

The Solution: Include Guards

#ifndef MATHUTILS_H  // If not defined
#define MATHUTILS_H  // Define it

// Your code here
int add(int a, int b);
int multiply(int a, int b);

#endif  // End if

How Include Guards Work

1

First Inclusion

#ifndef MATHUTILS_H  // MATHUTILS_H is NOT defined ‚Üí Condition TRUE
#define MATHUTILS_H  // Now define MATHUTILS_H

// This code RUNS
int add(int a, int b);

#endif
2

Second Inclusion (Same File)

#ifndef MATHUTILS_H  // MATHUTILS_H IS defined ‚Üí Condition FALSE
#define MATHUTILS_H  

// This code is SKIPPED
int add(int a, int b);

#endif

Result: No duplicate code, no errors!

Naming Convention for Include Guards

Best Practice:
  • Use ALL CAPS
  • Include filename in the guard name
  • Replace dots with underscores
  • Make it unique to avoid conflicts
File Name Include Guard
mathutils.h MATHUTILS_H
string_utils.h STRING_UTILS_H
graphics.engine.h GRAPHICS_ENGINE_H
myproject/utils.h MYPROJECT_UTILS_H

Complete Template

// filename.h
#ifndef FILENAME_H
#define FILENAME_H

// All your declarations and code here

class MyClass {
    // class definition
};

void myFunction();

#endif // FILENAME_H

üîπ Alternative: #pragma once

Modern compilers support a simpler alternative to include guards.

// mathutils.h
#pragma once

// Your code here
int add(int a, int b);
int multiply(int a, int b);

Comparison

Feature #ifndef Guards #pragma once
Lines of Code 3 lines 1 line
Compatibility 100% portable Most modern compilers
Speed Slightly slower Slightly faster
Error Prone Can have naming conflicts No naming issues
Recommendation:

Use #pragma once for modern projects. Use traditional guards for maximum portability or legacy systems.

üöÄ C++ Program - Complete Flow

When you write a C++ program, it goes through multiple stages before it becomes an executable. Understanding this flow is crucial for debugging errors and writing better code.

The Four Major Stages: 
Source Code ‚Üí Preprocessing ‚Üí Compilation ‚Üí Linking ‚Üí Execution

Let's Break Down Each Stage Microscopically

1

PREPROCESSING

The preprocessor is a program that runs before compilation begins. It modifies your source code.

What Happens in Preprocessing?

  • Header files are included (copied)
  • Macros are expanded
  • Comments are removed
  • Conditional compilation is processed

(A) Header File Expansion

#include <iostream>

What the preprocessor does:

It literally copies the ENTIRE content of the iostream file and pastes it at this location.

This is called: File Inclusion

The preprocessor replaces the #include line with thousands of lines from the header file!

(B) Macro Expansion

#define PI 3.14
#define SQUARE(x) ((x) * (x))

int main() {
    cout << PI;           // Becomes: cout << 3.14;
    cout << SQUARE(5);    // Becomes: cout << ((5) * (5));
}

Types of Macros:

  • Object-like macro: #define PI 3.14
  • Function-like macro: #define SQUARE(x) ((x) * (x))
This is called: Macro Expansion

Simple text replacement - wherever PI appears, it becomes 3.14

(C) Comments Removal

int x = 5;  // This is a comment
/* This is also a comment */

All comments are completely removed during preprocessing. The compiler never sees them!

(D) Conditional Compilation

#ifdef DEBUG
    cout << "Debug Mode Active" << endl;
#endif

#ifndef RELEASE
    cout << "Not in release mode" << endl;
#endif

Code inside these blocks is only compiled if the condition is true.

Output of Preprocessing:

A file with extension .i (e.g., main.i) is created. This is the expanded source code.

g++ -E main.cpp -o main.i  // Generate preprocessed file
2

COMPILATION

Now the actual compiler starts working on the preprocessed code.

What the Compiler Does:

  • Checks syntax (grammar rules)
  • Detects errors
  • Converts code to machine language
  • Performs optimizations

(A) Lexical Analysis - Tokenization

The code is broken down into smallest units called tokens.

int x = 5;

Tokens generated:

  • int ‚Üí Keyword
  • x ‚Üí Identifier
  • = ‚Üí Operator
  • 5 ‚Üí Literal
  • ; ‚Üí Delimiter

(B) Syntax Analysis (Parsing)

Checks if the grammar is correct according to C++ rules.

// Correct syntax:
int x = 5;

// ‚ùå Syntax Error:
int = x 5;  // Wrong order!

(C) Semantic Analysis

Checks if the code makes logical sense.

int x;
x = "hello";  // ‚ùå Semantic Error!
// Can't assign string to integer variable

(D) Intermediate Code Generation

Compiler creates an intermediate representation (often assembly-like code).

(E) Code Optimization

Removes unnecessary code and improves performance.

// Your code:
int x = 5;
int y = x + 0;  // Adding 0 is useless

// After optimization:
int x = 5;
int y = x;  // Optimizer removes + 0
Output of Compilation:

An Object File with extension .o or .obj is created (e.g., main.o).

g++ -c main.cpp -o main.o  // Generate object file

Important: The object file contains machine code, but it's NOT a complete executable program yet!

3

LINKING - The Critical Stage üî•

This is where everything comes together!

The Problem:

Your object file has machine code, but some functions are still missing!

#include <iostream>

int main() {
    std::cout << "Hello";  // Where is cout's actual code?
    return 0;
}

The cout function is declared in iostream, but its actual implementation is in the C++ standard library.

What the Linker Does:

  • Connects your object file with library files
  • Resolves external references (finds where functions are defined)
  • Combines multiple object files together
  • Creates the final executable
This is called: Symbol Resolution

The linker finds where each function/variable is actually defined and connects everything.

Example: Multi-File Linking

// mathutils.cpp
int add(int a, int b) {
    return a + b;
}

// main.cpp
#include "mathutils.h"
int main() {
    add(5, 3);  // Linker connects this call to mathutils.cpp
}

Common Linker Error: Undefined Reference

Error Message: 
undefined reference to `add(int, int)'

Reason: You declared the function but didn't compile the file containing its definition!

Solution: Compile all required .cpp files together

g++ main.cpp mathutils.cpp -o app

Types of Linking:

Type Description When Used
Static Linking Library code is copied into executable Larger file size, no external dependencies
Dynamic Linking Library loaded at runtime (.dll, .so files) Smaller file size, needs library at runtime
Output of Linking:

A fully executable program!

  • main.exe (Windows)
  • a.out or custom name (Linux/Mac)
g++ main.o mathutils.o -o app  // Link object files
4

EXECUTION

Now the CPU takes over!

Execution Steps:

  1. Loading: OS loads the program into RAM (memory)
  2. Memory Allocation: Stack and Heap memory are allocated
  3. Entry Point: main() function is called
  4. Instruction Execution: CPU executes instructions one by one
  5. Output: Results appear on screen or are saved to files
  6. Termination: Program exits, memory is freed

Memory Structure During Execution

When your program runs, memory is divided into 4 parts:

Memory Section Purpose Example
Stack Local variables, function calls Variables inside functions
Heap Dynamic memory allocation new, malloc
Data Segment Global & static variables Global variables
Code Segment Program instructions Compiled machine code

üß† Important Terms Explained

Token:

The smallest unit of code (e.g., int, x, =, 5, ;)

Identifier:

Name of a variable or function (e.g., myVariable, calculateSum)

Literal:

Direct value in code (e.g., 5, 3.14, "Hello", 'A')

Object File (.o, .obj):

Machine code that's not yet a complete program - needs linking

Executable File:

Fully linked and runnable program (.exe, a.out)

Symbol:

Name of a function or variable that the linker needs to resolve

üîπ Common Errors and Solutions

‚ùå Error 1: Undefined Reference

Error Message: 
undefined reference to `add(int, int)'

Cause: Function is declared in header but implementation file wasn't compiled.

Solution: Include ALL .cpp files in compilation:

// Wrong:
g++ main.cpp -o app

// Correct:
g++ main.cpp mathutils.cpp -o app

‚ùå Error 2: No Such File or Directory

Error Message: 
fatal error: mathutils.h: No such file or directory

Cause: Header file not found.

Solutions:

  • Make sure header file is in the same directory as source file
  • Use correct path: #include "subfolder/header.h"
  • Use angle brackets only for system headers: #include <iostream>
  • Use quotes for your headers: #include "mathutils.h"

‚ùå Error 3: Multiple Definition

Error Message: 
multiple definition of `add(int, int)'

Cause: Missing include guards or defining functions in header files.

Solution: Add include guards:

#ifndef MATHUTILS_H
#define MATHUTILS_H

// Your code

#endif

‚ùå Error 4: Redefinition

Error Message: 
redefinition of 'struct Point'

Cause: Header included multiple times without guards.

Solution: Always use include guards or #pragma once.

‚ùå Error 5: Wrong Include Syntax

// Wrong:
#include <mathutils.h>  // System path

// Correct:
#include "mathutils.h"  // Current directory

Debugging Checklist

When compilation fails, check:
  1. Are all .cpp files included in compilation command?
  2. Are header guards present in all .h files?
  3. Is the correct include syntax used (< > vs " ")?
  4. Are file paths correct?
  5. Are function declarations and definitions matching?

üî• Final Deep Concepts

Key Takeaways for Interviews

Header File = Interface

Tells other code what functions/classes are available, not how they work.

CPP File = Implementation

Contains the actual logic and function bodies.

#include = Copy Mechanism

Literally copies file contents before compilation.

Include Guard = Safety Lock

Prevents duplicate code from multiple inclusions.

Compilation = Convert

Transforms human-readable code to machine code.

Linking = Join

Connects multiple object files into one executable.

Best Practices Summary

  1. Always use include guards or #pragma once in header files
  2. Separate declaration and definition - declarations in .h, definitions in .cpp
  3. Use meaningful names for header guards (e.g., FILENAME_H)
  4. Include only what you need - avoid unnecessary includes
  5. Use angle brackets for system headers, quotes for your own
  6. Never put using namespace in headers - it pollutes global scope
  7. Document your headers with comments explaining purpose and usage
  8. Keep headers minimal - only declarations, not implementations

Advanced Tip: Forward Declarations

Sometimes you don't need to include a full header - just declare that a class exists:

// Instead of:
#include "BigClass.h"

// You can use:
class BigClass;  // Forward declaration

// Then include in .cpp file only where needed

This reduces compilation time and dependencies.

What You Should Never Do

Don't:
  • Define functions in header files (except inline/template)
  • Include .cpp files
  • Forget include guards
  • Use using namespace std; in headers
  • Create circular dependencies (A.h includes B.h, B.h includes A.h)
